# CDK 兑换系统 · 需求规格说明书

> 版本: v1.0  
> 日期: 2026-05-22  
> UI 原型: `index.html`(前台) / `admin.html`(后台) / `login.html`(后台登录)

---

## 一、项目概述

### 1.1 业务定位
一套与第三方卡网解耦的 **数字商品兑换平台**。管理员在后台维护商品与库存,批量生成卡密交付给卡网售卖;用户(买家)拿到卡密后在前台兑换,系统从库存中取出一条内容发给用户。

### 1.2 系统角色
| 角色 | 说明 | 是否登录 |
| --- | --- | --- |
| 终端用户(买家) | 在前台输入卡密兑换、查询卡密状态 | **不需要登录** |
| 管理员 | 维护商品、库存、卡密,查看数据 | 需要登录 |
| 卡网代理 | 不直接登录本系统,仅通过卡密兑换 | — |

### 1.3 核心特性
- 前台 **零登录** 直接兑换
- 卡密格式由系统 **写死**,不可自定义
- 库存内容 **一行一条、无字符长度限制**,支持超长字符串
- 前台兑换记录仅保存在 **用户本机浏览器**(localStorage),不进后端数据库
- 实时库存以 **5 格彩条** 显示,不暴露具体数字
- 与任何卡网解耦,**不预留外部卡网地址**(代理可自由对接)

## 二、技术架构建议

### 2.1 推荐技术栈
- **后端**: Node.js (Express/Koa/NestJS) 或 PHP (Laravel) 或 Go (Gin),按团队习惯
- **数据库**: MySQL 8.x / PostgreSQL 14+
- **缓存**: Redis(可选,用于库存计数 / 防刷限流)
- **前端**: 静态 HTML + 原生 JS(已交付),后端只提供 JSON API

### 2.2 部署形态
```
┌─────────────────┐         ┌─────────────────────┐
│  前台 index.html │ ──REST→ │   后端 API 服务      │
└─────────────────┘         │  (Node/Go/PHP ...)   │
                            ├─────────────────────┤
┌─────────────────┐         │     MySQL + Redis    │
│ 后台 admin.html  │ ──REST→ │                      │
└─────────────────┘         └─────────────────────┘
```

### 2.3 安全基线
| 项 | 要求 |
| --- | --- |
| 后台登录 | 账号+密码+图形验证码,密码 bcrypt/argon2 哈希,**禁止明文** |
| 接口鉴权 | 后台 API 全部走 JWT 或 Session,前台无需鉴权 |
| 兑换防刷 | 同 IP / Cookie 兑换间隔可配(默认 3s),触发限流返回 429 |
| 兑换并发 | 单条卡密兑换必须加 **分布式锁**(Redis SETNX 或 DB 行锁),避免一码多发 |
| 库存事务 | 兑换 = 卡密标记 + 库存出库,必须包在 **同一数据库事务** 中 |
| XSS / SQL 注入 | 所有用户输入参数化,前台输出 escape |

## 三、数据库设计

### 3.1 admin 表(管理员)
```sql
id            BIGINT PK
username      VARCHAR(64) UNIQUE
password_hash VARCHAR(255)
last_login_at DATETIME
last_login_ip VARCHAR(45)
created_at    DATETIME
```

### 3.2 product 表(商品)
```sql
id              BIGINT PK
name            VARCHAR(128)       -- 商品名称
intro           TEXT               -- 商品简介(兑换成功后展示)
usage_text      TEXT               -- 使用说明(兑换成功后展示)
cdk_prefix      VARCHAR(8)         -- 卡密前缀,3-5 位大写字母,如 BASE / PRO / VIP
full_threshold  INT DEFAULT 100    -- 满格阈值,库存≥此值显示 5 格绿色
status          TINYINT DEFAULT 1  -- 0 下架, 1 上架
sort            INT DEFAULT 0      -- 排序权重
created_at      DATETIME
updated_at      DATETIME
```
> **不存在 `category` 字段**,系统不做分类。

### 3.3 inventory 表(库存)
```sql
id          BIGINT PK
product_id  BIGINT FK product.id
content     LONGTEXT           -- 一条库存的完整内容(无长度限制,可含分隔符如 ----)
status      TINYINT DEFAULT 0  -- 0 未发货, 1 已发货, 2 作废
delivered_cdk_id BIGINT NULL   -- 发货时关联的卡密
expire_at   DATETIME NULL      -- 可选有效期
batch_no    VARCHAR(64)        -- 导入批次号(便于回滚)
created_at  DATETIME
delivered_at DATETIME NULL

INDEX (product_id, status)
INDEX (batch_no)
```
> **重要**: `content` 字段使用 `LONGTEXT`(MySQL)或 `TEXT`(PG),数据库前后端均不得对其做截断;前端文本框 `white-space: pre; overflow:auto`。

### 3.4 cdk 表(卡密)
```sql
id           BIGINT PK
code         VARCHAR(32) UNIQUE   -- PREFIX-XXXX-XXXX-XXXX-XXXX
product_id   BIGINT FK product.id
status       TINYINT DEFAULT 0    -- 0 未使用, 1 已使用, 2 已禁用, 3 已过期
valid_days   INT DEFAULT 0        -- 0 = 永久, >0 = 生成后 N 天过期
generated_at DATETIME
expire_at    DATETIME NULL        -- 根据 valid_days 计算; 0 时为 NULL
used_at      DATETIME NULL
used_ip      VARCHAR(45) NULL
inventory_id BIGINT NULL          -- 兑换后关联的库存
remark       VARCHAR(255)
batch_no     VARCHAR(64)

INDEX (product_id, status)
INDEX (status, expire_at)
INDEX (batch_no)
```

### 3.5 redeem_log 表(兑换/查询流水)
```sql
id          BIGINT PK
type        TINYINT          -- 1 兑换, 2 查询
cdk_code    VARCHAR(32)      -- 查询批量时此字段记原始数量摘要
product_id  BIGINT NULL
ip          VARCHAR(45)
ua          VARCHAR(255)
result      TINYINT          -- 1 成功, 2 卡密无效, 3 已使用, 4 已过期, 5 库存不足, 6 限流
created_at  DATETIME

INDEX (cdk_code)
INDEX (created_at)
```

### 3.6 admin_log 表(后台操作日志)
```sql
id        BIGINT PK
admin_id  BIGINT
action    VARCHAR(64)   -- import_inventory / gen_cdk / disable_cdk / edit_product ...
target    VARCHAR(255)  -- 描述操作对象
ip        VARCHAR(45)
created_at DATETIME
```

### 3.7 setting 表(站点配置)
KV 表:
```sql
key   VARCHAR(64) PK
value TEXT
```
预置键:
- `site_name` 默认 `兑换中心`
- `query_max` 默认 `50`
- `redeem_throttle_seconds` 默认 `3`
- `redeem_fail_text` 默认 `卡密无效或已使用`
- `frontend_history_keep` 默认 `200`(仅前端读取,后端只是值的存放点)
- `frontend_stock_visible` 默认 `1`
- `announcement` 公告富文本

## 四、卡密(CDK)规格

### 4.1 格式规则(**不可自定义**)
```
{PREFIX}-XXXX-XXXX-XXXX-XXXX
```
- `PREFIX`: 来自商品的 `cdk_prefix`,3-5 位大写字母
- `XXXX`: 4 段,每段 4 位
- 字符集: **大写字母 + 数字**,**排除易混字符** `I O L U 0 1`
  - 实际字符集: `ABCDEFGHJKMNPQRSTVWXYZ23456789`(28 字符)
- 总长度: 前缀 + `-` + 16 位字母数字 + 3 个 `-`

### 4.2 生成示例
```
BASE-FD5K-7XR3-Q9NM-CTYW
PRO-J3KH-5R8N-Z4VX-BPQA
VIP-MN6T-WX9R-A2HK-PJBN
```

### 4.3 有效期规则
- 生成时输入 **有效天数**(整数,默认 0)
  - `0` = 永久有效
  - `1` = 1 天后过期
  - `N` = N 天后过期
- 不使用日历日期选择器

### 4.4 唯一性
- `code` 字段建唯一索引,**生成时若碰撞需重试**(预期概率极低,28^16 ≈ 5×10^22)

## 五、前台(用户端)规格

### 5.1 页面结构
- **无 Logo、无站名、无顶部导航**
- **背景**: 紫色渐变 `linear-gradient(135deg, #6e5fda → #8068e0 → #7d6ce3)` + 3 个装饰圆光晕
- **主卡片**(居中,最大宽 760px):
  - Tab: `CDK 兑换` / `CDK 查询`
  - 兑换:单输入框 + 兑换按钮
  - 查询:textarea(每行一个,最多 50 条,实时计数 `n/50`)
- **历史卡片**(主卡片下方):
  - 表格列: 兑换时间 / 商品 / 卡密 / 详情按钮
  - 详情 → 弹窗显示完整商品内容
  - 「清空记录」按钮
- **库存悬浮卡**(右侧 `position: fixed`,< 1180px 自动落到下方):
  - 标题 `实时库存`
  - **无分类、无数字、扁平列表**
  - 每行: 商品名 + 5 个色块
  - 底部图例: 充足 / 偏少 / 紧缺 / 空

### 5.2 5 格库存色块算法
```js
ratio = min(stock / full_threshold, 1)
filled = stock === 0 ? 0 : max(1, ceil(ratio * 5))
颜色档位 = 
  filled >= 4 → 绿 #10b981 (充足)
  filled === 3 → 黄 #f59e0b (偏少)
  filled === 2 → 橙 #f87171 (较少)
  filled === 1 → 红 #ef4444 (紧缺)
  filled === 0 → 全灰 #e5e7eb (售罄)
```
- 5 个 `<i>` 标签水平排列,填充 `filled` 个色块,其余保持灰色

### 5.3 兑换交互
1. 用户输入卡密 → 点 `兑换卡密`
2. 后端校验:
   - 卡密格式 → 卡密存在 → 未使用 → 未过期 → 未禁用 → 商品上架 → 库存非空
3. 任一不通过 → 返回错误码 + 提示(使用 `redeem_fail_text`)
4. 通过 → 事务内:
   - 锁定一条库存(`SELECT ... FOR UPDATE` 或 Redis 锁)
   - 标记 `cdk.status = 1`、`inventory.status = 1`
   - 写 `redeem_log`
5. 返回 `{ product: {name, intro, usage_text}, content: "<库存原文>", time: "..." }`
6. 前端弹出成功块 + 自动写入 `localStorage`

### 5.4 查询交互
1. 用户粘贴最多 50 条卡密 → 点 `批量查询`
2. 后端遍历:
   - 不存在 → `invalid`
   - `status=0 且 未过期` → `unused`
   - `status=1` → `used`
   - 已禁用/过期 → `expired`
3. **不返回任何商品内容**,只返回每条卡密的状态字段
4. 前端按状态展示徽章

### 5.5 localStorage 数据结构
```js
key: 'cdk_redeem_history_v1'
value: [
  {
    time: '2026-05-22 14:36:12',
    product: '基础成品号',
    cdk: 'BASE-XXXX-XXXX-XXXX-XXXX',
    intro: '...',
    usage: '...',
    content: '完整库存原文(可含 ----、换行等)'
  },
  ...
]
```
- 最大长度 `setting.frontend_history_keep`(默认 200),超出从尾部弃
- 仅前端读写,**后端无任何接口涉及前端历史**

### 5.6 限制清单
| 项 | 数值 |
| --- | --- |
| 单次兑换数量 | 1(不允许批量) |
| 单次查询数量 | 50(可在站点设置调整) |
| 兑换防刷间隔 | 默认 3s,可在站点设置调整 |
| 输入框字符上限 | 32(够格式长度即可) |

## 七、API 接口设计

> 所有接口走 RESTful,响应体统一:
> ```json
> { "code": 0, "msg": "ok", "data": { ... } }
> ```
> `code: 0` 成功,其余为错误码。

### 7.1 前台公开接口(无需鉴权)

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/redeem` | 兑换卡密 `{ cdk: "xxx" }` → `{ product, content, time }` |
| POST | `/api/query` | 批量查询 `{ cdks: ["a","b",...] }` → `[{ cdk, status }]` |
| GET | `/api/stocks` | 获取实时库存档位 `[{ product_id, name, full_threshold, stock_filled(0-5) }]` |
| GET | `/api/site` | 公开站点配置 `{ announcement, stock_visible }` |

### 7.2 后台接口(JWT 鉴权)

| 模块 | 路径 |
| --- | --- |
| 登录 | `POST /admin/login` / `POST /admin/logout` |
| 商品 | `GET/POST /admin/products` `PUT/DELETE /admin/products/:id` |
| 库存 | `POST /admin/inventory/import` `GET /admin/inventory` `DELETE /admin/inventory/:id` |
| 卡密 | `POST /admin/cdks/generate` `GET /admin/cdks` `POST /admin/cdks/disable` `GET /admin/cdks/export` |
| 兑换记录 | `GET /admin/redeem-logs` |
| 设置 | `GET/PUT /admin/settings` `PUT /admin/password` |
| 操作日志 | `GET /admin/admin-logs` |
| 仪表盘 | `GET /admin/dashboard/summary` `GET /admin/dashboard/trend?days=14` |

| code | 含义 |
| --- | --- |

| 1003 | 卡密已禁用 |

## 八、关键业务流程

### 8.1 兑换流程(伪代码)
```text
POST /api/redeem { cdk }
  1. 限流: redis incr "redeem:ip:{ip}" expire 3s, > 1 则返回 1007
  2. 校验格式: 正则 /^[A-Z]{3,5}(-[A-Z2-9]{4}){4}$/
  3. SELECT cdk WHERE code = ? FOR UPDATE
  4. 状态校验 (status / expire_at)
  5. SELECT product WHERE id = ? AND status = 1
  6. SELECT inventory WHERE product_id=? AND status=0 LIMIT 1 FOR UPDATE
  7. UPDATE cdk SET status=1, used_at=now, used_ip=?, inventory_id=?
  8. UPDATE inventory SET status=1, delivered_cdk_id=?, delivered_at=now
  9. INSERT redeem_log (type=1, result=1, ...)
 10. COMMIT
 11. 返回 { product, content }
```

### 8.2 生成卡密流程
```text
POST /admin/cdks/generate { product_id, count, valid_days, remark }
  1. 查商品,拿到 cdk_prefix
  2. 循环 count 次:
       code = `${prefix}-${rand4()}-${rand4()}-${rand4()}-${rand4()}`
       去重(内存 Set + INSERT IGNORE 后 affected_rows 校验)
  3. 批量 INSERT
  4. expire_at = valid_days > 0 ? now + valid_days*86400 : NULL
  5. 写 admin_log
  6. 返回完整 code 列表,前端用 textarea + 复制按钮呈现
```

### 8.3 库存导入流程
```text
POST /admin/inventory/import { product_id, mode, contents[], expire_at? }
  1. 校验模式: append / overwrite / dedupe
  2. overwrite: DELETE WHERE product_id AND status=0
  3. dedupe: 用 hash(content) 过滤已存在
  4. 批量 INSERT,batch_no = now timestamp
  5. 写 admin_log
```

  - 背景 `#6e5fda → #8068e0 → #7d6ce3`
- 卡片: 白色 + `border-radius: 16px` + 紫色调阴影

- 表格行 hover `#fafbff`

### 9.3 字体
- 中文: `PingFang SC` / `Microsoft YaHei`
- 等宽(卡密/库存): `ui-monospace, Menlo, Consolas, monospace`

### 9.4 响应式

- 后台 < 1200px 时统计卡 6 列变 3 列,< 760px 变单列
